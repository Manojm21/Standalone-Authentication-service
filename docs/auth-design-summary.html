Auth System — Complete Design Summary

Project: FastAPI Backend  |  Phase 2 Complete  |  June 2026

1. Token Comparison

Access Token Refresh Token
Format JWT (signed, base64-encoded JSON) Opaque random string (secrets.token_urlsafe(32))
Lifetime 15 minutes 7 days
Client Storage Memory (sent in Authorization: Bearer header) App storage (sent in request body to /refresh and /logout)
Server Storage Nowhere (stateless) SHA-256 hash in refresh_tokens table
Validation Signature + expiry check (no DB hit) Hash lookup in DB
Instantly Revocable? No — must wait for 15-min expiry Yes — set revoked = True
JWT Payload {"sub": "user_id", "exp": timestamp, "type": "access"} N/A — no information inside (random bytes)

2. Datetime Handling (SQLite Quirk)

WhenWhat HappensTimezone
Writing to DB datetime.now(timezone.utc) Aware (has UTC info)
SQLite stores it Strips timezone → "2026-06-18 15:30:00" Naive (no timezone)
Reading from DB SQLAlchemy returns naive datetime Naive
Comparing .replace(tzinfo=timezone.utc) < datetime.now(timezone.utc) Convert back to aware

Note: This problem disappears with PostgreSQL (Phase 5) — Postgres handles timezone-aware datetimes natively.

3. Execution Flows

3.1 Register

Client                          Server                         DB
  │                               │                             │
  ├─ POST /register ─────────────►│                             │
  │  {email, password}            │─ hash_password(bcrypt) ────►│
  │                               │─ INSERT users ─────────────►│
  │◄─ 201 {id, email} ───────────│                             │

3.2 Login (creates both tokens)

Client                          Server                         DB
  │                               │                             │
  ├─ POST /login ────────────────►│                             │
  │  {username, password}         │─ SELECT user by email ─────►│
  │                               │─ bcrypt.checkpw()           │
  │                               │─ create_access_token(JWT)   │ (no DB write)
  │                               │─ create_refresh_token()     │
  │                               │─ sha256(raw) ──────────────►│ INSERT refresh_tokens
  │◄─ 200 {access, refresh} ─────│                             │
  │                               │                             │
  │  Client saves BOTH tokens     │                             │

3.3 Authenticated Request (/me or any protected route)

Client                          Server                         DB
  │                               │                             │
  ├─ GET /me ────────────────────►│                             │
  │  Header: Bearer <access>      │─ jwt.decode()              │ (NO DB hit)
  │                               │─ check type == "access"    │
  │                               │─ check not expired         │
  │                               │─ SELECT user by id ────────►│
  │◄─ 200 {id, email} ───────────│                             │

3.4 Refresh (with rotation)

Client                          Server                         DB
  │                               │                             │
  ├─ POST /refresh ──────────────►│                             │
  │  {refresh_token: raw}         │─ sha256(raw)               │
  │                               │─ SELECT by hash ───────────►│
  │                               │─ check not revoked          │
  │                               │─ check expires_at > now     │
  │                               │─ old.revoked = True ───────►│ UPDATE (rotation)
  │                               │─ create new refresh token   │
  │                               │─ INSERT new row ───────────►│
  │                               │─ create new access token    │
  │◄─ 200 {new_access, new_refresh}                            │
  │                               │                             │
  │  OLD refresh token is DEAD    │                             │

3.5 Logout (single device only)

Client                          Server                         DB
  │                               │                             │
  ├─ POST /logout ───────────────►│                             │
  │  {refresh_token: raw}         │─ sha256(raw)               │
  │                               │─ SELECT by hash ───────────►│
  │                               │─ row.revoked = True ───────►│ UPDATE
  │◄─ 204 No Content ────────────│                             │
  │                               │                             │
  │  Client discards access token │                             │

4. When Does Authorization Die?

ScenarioAccess TokenRefresh TokenUser Must...
Normal expiry Dies after 15 min Dies after 7 days Refresh → or → Re-login
Refresh called Still alive (until own 15min) OLD one dies immediately (rotation) Use new tokens
Logout called Still works up to 15 min Dies immediately Re-login when access expires
Token stolen + rotation happened Attacker's access works briefly Attacker's refresh is dead Attacker locked out after 15 min
Both tokens lost (app crash) Can't send it Can't send it Re-login

The 15-Minute Window Trade-off

After logout, the access token remains valid for up to 15 minutes. This is the deliberate trade-off of stateless JWTs — checking every access token against a DB on every request would defeat the purpose. If you ever need instant access-token kill, that's what Redis denylists solve (Phase 5).

5. Error Contract (Frontend Decision Tree)

ResponseMeaningClient Action
401 {"detail": "expired_token"} Access token was valid but time's up → Call POST /refresh
401 {"detail": "invalid_token"} Token is garbage, tampered, or wrong type → Redirect to login
401 {"detail": "Invalid or expired refresh token"} Refresh token dead/expired/revoked → Redirect to login
204 on /logout Logout processed (or token was already dead) → Clear local tokens

6. Multi-Device Model

User "test@test.com"
  ├── Login from Laptop  → RefreshToken row #1 (jti: abc...)
  ├── Login from Phone   → RefreshToken row #2 (jti: def...)
  └── Login from Tablet  → RefreshToken row #3 (jti: ghi...)

Logout Phone → ONLY row #2 revoked
Laptop and Tablet continue working independently

Each login creates its own row. Each row has its own lifecycle. They never interfere.

7. Security Layers

LayerHow
Password never stored rawbcrypt hash (salt + slow hash)
Refresh token never stored rawSHA-256 hash (DB leak = useless)
Access token can't be forgedHMAC-SHA256 signature with SECRET_KEY
Refresh can't be used as access"type": "access" claim checked in get_current_user
Old refresh can't be reusedRotation sets revoked = True on every refresh
SECRET_KEY validated at bootpydantic Settings with min_length=32 — app won't start without it
